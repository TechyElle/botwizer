🔮 Pillar 1 of 4

Abstraction

Hiding implementation details behind a strict interface contract

bot_data.py — TargetingAction (Abstract Base Class) 
# Import the ABC tools from Python's standard library
from abc import ABC, abstractmethod


class TargetingAction(ABC):
    """ Base class representing a targeting action (Abstraction) """

    @abstractmethod
    def process_target(self, target_name):
        """ Process target name (abstract method) """
        pass    # ← No implementation here!


# ❌ This would FAIL — cannot instantiate abstract class:
# action = TargetingAction()  # → TypeError!

# ✅ Verified by unit test:
with self.assertRaises(TypeError):
    TargetingAction()   # → confirms abstraction enforced
🔮 What is Abstraction?
Abstraction means exposing only what is necessary and hiding the internal details. In Python, this is achieved using the abc module — ABC (Abstract Base Class) and @abstractmethod.
📌 In Botwizer
TargetingAction is the abstract base class. It declares the method process_target() but does NOT implement it (pass). Any class that inherits from it must override this method — otherwise Python will raise a TypeError.
ABC @abstractmethod TargetingAction process_target
✅ Why It Matters
The caller of process_target() does not need to know how it works — only that it works. This is the essence of abstraction: define the "what", hide the "how".
🧬 Pillar 2 of 4

Inheritance

Building a class hierarchy to reuse code and share behavior

bot_data.py — HashtagTargeter & AccountTargeter (Subclasses) 
# HashtagTargeter inherits from TargetingAction
class HashtagTargeter(TargetingAction):
    """ Targets posts under a specific hashtag (Inheritance) """

    def process_target(self, target_name):
        """ Polymorphic implementation for hashtag """
        try:
            print(f"Searching hashtag: #{target_name}")
            print(f"Liking most recent post")
            return True
        except Exception as error:
            return False


# AccountTargeter also inherits from TargetingAction
class AccountTargeter(TargetingAction):
    """ Targets posts of a specific account (Inheritance) """

    def process_target(self, target_name):
        """ Polymorphic implementation for account """
        try:
            print(f"Visiting profile: @{target_name}")
            print(f"Following user: @{target_name}")
            return True
        except Exception as error:
            return False


# ✅ isinstance() confirms inheritance chain:
isinstance(HashtagTargeter(), TargetingAction)  # → True
isinstance(AccountTargeter(), TargetingAction)  # → True
🧬 What is Inheritance?
Inheritance allows a class (child) to acquire properties and behaviors from another class (parent), promoting code reuse and establishing an IS-A relationship.
📌 In Botwizer — Hierarchy Diagram
TargetingAction
«abstract» (ABC)
+ process_target(target_name) «abstract»
HashtagTargeter
+ process_target() ← #hashtag
AccountTargeter
+ process_target() ← @account
✅ Key Benefit
Both subclasses share the same type (TargetingAction) — proven by isinstance() checks in the unit tests. This makes them interchangeable in the runner.
HashtagTargeter AccountTargeter TargetingAction IS-A relation

Polymorphism

One method call — many different behaviors depending on the runtime object

run_bot.py — Polymorphic dispatch (L71–L88) 
# STEP 1: Instantiate the correct subclass based on user input
if self.target_type == "1":
    targeter = HashtagTargeter()     # ← assigned at runtime
    print("Initializing hashtag automation...")
else:
    targeter = AccountTargeter()     # ← or this one
    print("Initializing account automation...")


# STEP 2: The SAME call works for BOTH subclasses ✨
print("--- Processing Targets ---")
for target_item in self.session.get_targets():
    print(f"Processing target: {target_item}")

    success = targeter.process_target(target_item) # ← polymorphic!

    if success:
        print(f"✅ Processed: {target_item}")
    else:
        print(f"❌ Failed: {target_item}")


# RESULT: Same line of code → different behavior:
# HashtagTargeter → prints "Searching hashtag page: #python"
# AccountTargeter → prints "Visiting user account: @jack"
🔄 What is Polymorphism?
"Poly" = many, "morph" = form. Polymorphism allows one interface to be used for a general class of actions. The specific action is determined at runtime, not compile time.
📌 In Botwizer
The key line is: targeter.process_target(target_item)

The variable targeter can be either a HashtagTargeter or an AccountTargeter depending on user input. The same line of code produces completely different output at runtime.
runtime dispatch process_target() override
✅ Verified by Test
test_inheritance_and_polymorphism() in the unit test file calls process_target() on both subclasses and asserts both return True — confirming the polymorphic behavior works correctly with different internal implementations.
🔒 Pillar 4 of 4

Encapsulation

Protecting internal data from unauthorized access using private fields and validated accessors

bot_data.py — BotSession (Encapsulation) 
class BotSession:
    """ Manages user session and targets (Encapsulation) """

    def __init__(self):
        # Private attributes — double underscore = name mangled
        self.__username = ""     # ← can't access from outside!
        self.__password = ""     # ← protected by Python
        self.__targets  = []     # ← not directly modifiable
        self.__comments = []

    # ── GETTER (read access) ──────────────────────
    def get_username(self):
        return self.__username

    # ── SETTER (validated write access) ──────────
    def set_username(self, username_value):
        try:
            if not username_value or len(username_value.strip()) == 0:
                raise ValueError("Username cannot be empty")
            self.__username = username_value.strip()
        except ValueError as error:
            raise ValueError(f"Invalid username: {error}")

    def set_password(self, password_value):
        try:
            if len(password_value.strip()) < 6:    # ← validation!
                raise ValueError("Password ≥ 6 chars")
            self.__password = password_value.strip()
        except ValueError as error:
            raise ValueError(f"Invalid password: {error}")
🔒 What is Encapsulation?
Encapsulation bundles data and behavior into a class and restricts direct access to internal state. Only designated methods (getters/setters) can interact with private data.
📌 In Botwizer — Private Fields
BotSession uses Python's double underscore name mangling:
  • __username → mangled to _BotSession__username
  • __password → validates min 6 chars in setter
  • __targets → only addable via add_target()
  • __comments → only addable via add_comment()
__private getter setter validation
✅ What Happens if You Bypass It?
session.__password → AttributeError! You can only use session.get_password() to read and session.set_password("secure123") to write — with full validation.

Unit Tests — Proving OOP Works

All four OOP pillars are verified through automated unit tests

test_oop_implementation.py 
import unittest
from botwizer_automation.bot_data import (
    BotSession, HashtagTargeter, AccountTargeter, TargetingAction
)


class TestOopAutomation(unittest.TestCase):

    # ── Test: ABSTRACTION ─────────────────────
    def test_abstraction(self):
        # Cannot instantiate abstract class directly
        with self.assertRaises(TypeError):
            TargetingAction()   # → MUST raise TypeError ✅

    # ── Test: INHERITANCE + POLYMORPHISM ──────
    def test_inheritance_and_polymorphism(self):
        hashtag = HashtagTargeter()
        account = AccountTargeter()
        # Verify inheritance chain
        self.assertTrue(isinstance(hashtag, TargetingAction))  # ✅
        self.assertTrue(isinstance(account, TargetingAction))  # ✅
        # Verify polymorphic behavior returns True
        self.assertTrue(hashtag.process_target("coding"))      # ✅
        self.assertTrue(account.process_target("jack_coder"))  # ✅

    # ── Test: ENCAPSULATION ───────────────────
    def test_encapsulation_password(self):
        session = BotSession()
        session.set_password("mypassword123")  # valid ✅
        self.assertEqual(session.get_password(), "mypassword123")
        with self.assertRaises(ValueError):
            session.set_password("12345")    # too short → ValueError ✅
🧪 What the Tests Prove
  • Abstraction: TargetingAction() raises TypeError
  • Inheritance: isinstance(hashtag, TargetingAction) == True
  • Polymorphism: Both subclasses return True from same method signature
  • Encapsulation: Short password raises ValueError
