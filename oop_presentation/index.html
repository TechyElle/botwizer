🔮 Pillar 1 · Abstraction
bot_data.py 
# abc = Abstract Base Class module
from abc import ABC, abstractmethod


class TargetingAction(ABC):
    """ Abstract base class for all targeting actions """

    @abstractmethod
    def process_target(self, target_name):
        pass  # ← No implementation — MUST be in subclass


# ❌ Cannot instantiate abstract class:
# TargetingAction() → TypeError!

# ✅ Unit test confirms abstraction:
with self.assertRaises(TypeError):
    TargetingAction()  # PASSES ✔

Abstraction

Hides implementation details and defines a strict interface contract.

🔑 Key Concept

  • ABC = Abstract Base Class from Python's abc module
  • @abstractmethod forces every subclass to implement process_target()
  • Direct instantiation → TypeError — Python enforces it

💡 In Plain English

Define what to do, not how to do it. Callers only see the interface — never the implementation.

🧬 Pillar 2 · Inheritance
bot_data.py 
# HashtagTargeter inherits from TargetingAction
class HashtagTargeter(TargetingAction):
    def process_target(self, target_name):
        print(f"Searching hashtag: #{target_name}")
        print(f"Liking recent post under #{target_name}")
        return True


# AccountTargeter also inherits from TargetingAction
class AccountTargeter(TargetingAction):
    def process_target(self, target_name):
        print(f"Visiting profile: @{target_name}")
        print(f"Following user: @{target_name}")
        return True


# ✅ Both ARE a TargetingAction (IS-A relationship):
isinstance(HashtagTargeter(), TargetingAction)  # True
isinstance(AccountTargeter(), TargetingAction)  # True

Inheritance

Builds a hierarchy so subclasses reuse and extend base class behavior.

TargetingAction
«abstract» (ABC)
+ process_target()
HashtagTargeter
#hashtag logic
AccountTargeter
@account logic

💡 Key Benefit

Both subclasses share the same type — making them interchangeable wherever a TargetingAction is expected.

🔄 Pillar 3 · Polymorphism
run_bot.py — L72–L88 
# Step 1 — pick subclass based on user's choice
if self.target_type == "1":
    targeter = HashtagTargeter()   # ← runtime choice A
else:
    targeter = AccountTargeter()   # ← runtime choice B


# Step 2 — the SAME call, different result each time
for target_item in self.session.get_targets():

    success = targeter.process_target(target_item)
    #          ↑ same line of code ↑
    # If HashtagTargeter → searches #hashtag
    # If AccountTargeter → visits @profile

    if success:
        print(f"✅ Done: {target_item}")

Polymorphism

One method signature → many different behaviors depending on the object at runtime.

🔑 The Magic Line

This single line in the runner:

targeter.process_target(target_item)

…produces completely different output depending on which subclass targeter holds — without the runner caring which one it is.

💡 Poly = Many  ·  Morph = Form

The caller never checks the subclass type. Behavior is determined at runtime.

🔒 Pillar 4 · Encapsulation
bot_data.py — BotSession 
class BotSession:
    def __init__(self):
        self.__username = ""  # private (__)
        self.__password = ""  # private (__)
        self.__targets  = []
        self.__comments = []

    # Getter — read-only access
    def get_username(self):
        return self.__username

    # Setter — validated write access
    def set_password(self, pwd):
        if len(pwd.strip()) < 6:
            raise ValueError("Password ≥ 6 chars")
        self.__password = pwd.strip()

# ❌ Direct access FAILS:
# session.__password  → AttributeError!

Encapsulation

Bundles data and behaviour together. Shields private state from direct external access.

🔑 Python Double Underscore

  • __username → name-mangled to _BotSession__username
  • Can't be accessed with session.__username from outside
  • Only reachable via get_username() and set_username()

💡 Why Validation Matters

The setter rejects a password shorter than 6 characters — data integrity is enforced at the boundary, not scattered throughout the codebase.
